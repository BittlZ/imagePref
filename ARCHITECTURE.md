# Архитектура решения

Документ описывает, **что** мы строим, **почему** именно так и **как** это
исполняется на Kaggle (2× T4). Метрика соревнования — **ROC AUC**, то есть важен
только ранжирующий порядок предсказаний, а не их калибровка.

---

## 1. Постановка задачи

Дана пара сгенерированных изображений `(image_1, image_2)`; нужно предсказать
вероятность того, что **`image_1` лучше** (`is_image1_better`). По сути это
**reward-модель человеческого/экспертного предпочтения** между двумя картинками,
**без текста промпта** — сравниваем чисто по изображениям.

### Что показал EDA
| Наблюдение | Значение | Следствие для архитектуры |
|---|---|---|
| Размер train / test | 8710 / 4290 пар | Мало данных → опора на претрейн, борьба с оверфитом |
| Разрешение | всегда 1000×1000 RGB | Ресайз под бэкбон; деталь важна для «качества» |
| `P(is_image1_better=1)` | 0.347 | Позиционный приор (`image_2` чаще лучше) |
| Формат `image_1` / `image_2` | JPEG/PNG/WEBP / **всегда WEBP** | Слоты асимметричны по кодеку → добавляем как фичу + swap-аугментация |
| Утечка по размеру байтов / простым метрикам | AUC ≈ 0.50–0.54 | Дешёвых утечек нет — нужна реальная vision-модель |

**Главные принципы дизайна, вытекающие из EDA:**
1. **Замороженный сильный претрейн + лёгкая голова** — устойчиво к оверфиту на 8710 примерах.
2. **Bradley–Terry формулировка** — антисимметрия «кто лучше» встроена в модель и прямо оптимизирует ранкинг.
3. **Ансамбль разных индуктивных байесов** (CNN + CLIP) — стабильно поднимает AUC.
4. **Метаданные-приор** (формат/размер) скармливаем GBDT — пусть сам решит их полезность.

## 2. Общая схема (data flow)

```
parquet (image bytes + label)
        │
        ▼
[ features.py ]  декод → ресайз/нормализация → timm backbone (frozen)
        │        + hflip TTA, L2-norm           → эмбеддинг f на картинку
        │  кеш .npy:  F1, F2 (train) ; F1te, F2te (test)
        ▼
   ┌─────────────────────────────┬───────────────────────────────┐
   ▼                             ▼                               ▼
[ bt_head.py ]            [ pair_features.py → gbdt.py ]   [ finetune.py ] (опц.)
Bradley–Terry MLP         LightGBM на [diff, prod,         end-to-end сиамский
logit = s(f1) − s(f2)      |diff|, cos, meta]              backbone, logit=s1−s2
   │                             │                               │
   └───────────── OOF предсказания каждого компонента ───────────┘
                                 │
                                 ▼
                    [ cv.py ] поиск весов блендинга по OOF (rank-scale)
                                 │
                                 ▼
                    [ submit.py ] submission.csv (rank-normalized score)
```

## 3. Компоненты

### 3.1 Данные (`data.py`)
- Parquet хранит сырые байты картинок в двух бинарных колонках + int-метка; файл
  большой и одно-row-group, поэтому байты колонки читаем **один раз в память**
  (на Kaggle GPU-нодах ~29 GB RAM хватает) и **декодируем лениво** в `Dataset` —
  это даёт случайный доступ для `DataLoader`.
- `read_label_and_meta` дополнительно собирает **метаданные**: размеры байтов и
  контейнерный формат каждого слота — дешёвый позиционный/кодековый приор для GBDT.
- `PairImageDataset` для файнтюна умеет **swap-аугментацию** (меняет местами
  картинки и инвертирует метку) — это и есть антисимметрия Bradley–Terry.

### 3.2 Извлечение признаков (`features.py`)
- Любой бэкбон из `timm` (`num_classes=0` → чистый эмбеддинг). Трансформ берётся
  из `resolve_model_data_config`, что гарантирует правильную нормализацию/размер
  под конкретные веса. `img_size` можно поднять (деталь важна для «качества»).
- **TTA hflip**: усредняем эмбеддинг картинки и её отражения за один проход.
- **L2-нормализация** эмбеддингов → согласованные масштабы для голов и косинуса.
- **2× T4**: модель оборачивается в `DataParallel`, инференс идёт в **AMP fp16**.
- **Кеш**: результат для каждой пары (бэкбон, split, колонка) сохраняется в `.npy`;
  повторный запуск пропускает уже посчитанное.

### 3.3 Сравнительные фичи (`pair_features.py`)
Из эмбеддингов пары строим «контраст»:
- `compact` (по умолчанию): `[F1−F2, F1*F2, |F1−F2|, cos(F1,F2), meta]` — антисимметрично.
- `full`: то же + сырые `F1, F2`.
- `meta`: one-hot формата `image_1`, `log` размеров байтов, их разность/отношение.

### 3.4 Головы (`models/`)

**Bradley–Terry (`bt_head.py`)** — основная нейросетевая голова:
- Общий MLP `s(·)`: `Linear → BN → GELU → Dropout` (×N) → скаляр.
- `logit = s(f1) − s(f2)`, лосс `BCEWithLogits`.
- На каждом шаге **swap-аугментация** (с вероятностью 0.5 меняем местами → инвертируем метку).
- AdamW + cosine LR, **early stopping по валидационному AUC**, лучшее состояние восстанавливается.
- Антисимметрия встроена; `s(f)` — абсолютная «оценка качества» картинки.

**LightGBM (`gbdt.py`)** — табличная голова:
- Бустинг на сравнительных фичах; регуляризация (`feature_fraction`, `lambda_l2`,
  `min_child_samples`), early stopping по AUC. Классически силён и отлично блендится с MLP.

**End-to-end файнтюн (`finetune.py`, опционально)** — путь с высоким потолком:
- Сиамский backbone (shared) + голова → скаляр; `logit = s1 − s2`.
- Аугментации (RandomResizedCrop, hflip, лёгкий ColorJitter), **freeze-эпохи** в начале,
  раздельный LR для backbone/головы, **AMP fp16**, `DataParallel` на 2 GPU.
- Запускается независимо и блендится с фростовыми головами.

### 3.5 Кросс-валидация и блендинг (`cv.py`)
- **5-fold StratifiedKFold** по метке → честный **OOF AUC** (главный и единственный
  надёжный ориентир; паблик-лидерборд мал).
- Блендинг на **ранговой шкале** (устойчив к разным диапазонам скорингов).
- `search_blend_weights` — перебор выпуклых весов по OOF, выбор максимума AUC.

### 3.6 Оркестрация (`pipeline.py`) и сабмит (`submit.py`)
- `run_features` → `run_train` (CV всех голов + блендинг, сохраняет OOF/test/веса/`report.json`)
  → `run_predict` (блендит закешированные test-предсказания и пишет `submission.csv`).
- В сабмит идёт **ранк-нормализованный скор** как псевдо-вероятность (для ROC AUC калибровка не нужна).

## 4. Конфигурация (`config.py` + `configs/default.yaml`)
Все гиперпараметры вынесены в **один YAML-файл** `configs/default.yaml`. Порядок
сборки: дефолты `Config` → автодетект путей Kaggle → значения из YAML/JSON →
kwargs-оверрайды. `auto_config()` находит `/kaggle/input/.../train.parquet` и
выводит результат в `/kaggle/working`; локально пути указывают на `data/`. Любой
путь, заданный в файле, побеждает автодетект. Ключевые ручки: `backbones`,
`use_tta_hflip`, `use_gbdt`/`gbdt_pair_mode`, `finetune_enabled`,
`n_folds`/`seed`/`blend_search`, параметры BT-головы и LightGBM.

**Бэкбоны по умолчанию (ансамбль):**
| Имя (`timm`) | Тип | Зачем |
|---|---|---|
| `convnextv2_base.fcmae_ft_in22k_in1k_384` | CNN @384 | текстура, артефакты генерации, мелкая деталь |
| `vit_large_patch14_clip_224.openai` | CLIP ViT-L | семантика, общий «вкус» предпочтения |

(Кандидаты для расширения ансамбля: SigLIP/SigLIP2, DINOv2 ViT-L, EfficientNet-V2.)

## 5. Почему это должно выигрывать
- **Под малый датасет**: замороженные фичи + лёгкие головы почти не переобучаются.
- **Под метрику**: Bradley–Terry и ранговый блендинг прямо оптимизируют порядок (AUC).
- **Под железо (2× T4)**: тяжёлый этап (фичи) считается один раз и кешируется; головы
  обучаются за секунды; файнтюн использует обе карты через `DataParallel` + fp16.
- **Запас по потолку**: ансамбль разных бэкбонов + опциональный end-to-end файнтюн,
  блендинг по честному OOF.

## 6. План итераций
1. **Бейзлайн**: один бэкбон (frozen) + BT-голова + 5-fold CV + рабочий сабмит — опорный AUC.
2. **+ LightGBM** на сравнительных фичах и **+ второй бэкбон** → ансамбль/блендинг.
3. **End-to-end файнтюн** лучшего бэкбона, добавление в блендинг — дожать потолок.
4. **TTA / больший `img_size` / новые бэкбоны** — финальная докрутка.
