# Image Preference — соревнование по предсказанию лучшей картинки

Решение для задачи **«какая из двух сгенерированных картинок лучше»** (метрика — **ROC AUC**).

Подробная архитектура — в [`ARCHITECTURE.md`](ARCHITECTURE.md).

---

## Введение

Современные генеративные модели способны выдать по одному промпту десятки правдоподобных
изображений — но какое из них человек назовёт «лучшим»? Именно эту субъективную оценку
качества и пытается формализовать соревнование: научить модель предсказывать выбор экспертной
комиссии между двумя генерациями. Это близко к тому, как работают reward-модели
(PickScore, ImageReward, HPS), обучающие предпочтения человека для последующей настройки
генеративных пайплайнов.

Практическая сложность в том, что данных мало (8710 обучающих пар), промпт недоступен, а
«качество» часто прячется в мелких деталях — артефактах генерации, резкости, композиции.
Поэтому обучать тяжёлую сеть с нуля бессмысленно: выигрывает опора на сильные предобученные
представления и аккуратная работа с ранжированием пар. Ниже — как мы к этому пришли и что
получилось.

---

## Задача

На входе пара изображений `(image_1, image_2)` — две генерации (часто по одному промпту, но **без текста промпта** в данных). Нужно предсказать вероятность того, что **`image_1` признана лучше** экспертной комиссией (`is_image1_better`).

| | train | test |
|---|---|---|
| Пар | 8710 | 4290 |
| Разрешение | 1000×1000 RGB | 1000×1000 RGB |
| Формат `image_1` | JPEG / PNG / WEBP | то же |
| Формат `image_2` | **всегда WEBP** | **всегда WEBP** |
| Баланс метки | `P(image_1 лучше) ≈ 0.35` | — |

По сути это **reward-модель человеческого предпочтения** между двумя картинками: важен не абсолютный скор «красоты», а **ранжирование** внутри пары (именно это оптимизирует ROC AUC).

---

## Наш подход

Двухэтапный пайплайн: тяжёлое извлечение признаков один раз на GPU, обучение голов — быстро на CPU из готовых эмбеддингов.

```
parquet (байты картинок)
        │
        ▼  [GPU] frozen timm-бэкбоны + hflip TTA → L2-норм эмбеддинг f на картинку
        │  сохраняем в .npy (по колонке: image_1 / image_2, train / test)
        ▼
   ┌────────────────────────────────────────────────────────────┐
   │  [CPU] для каждого бэкбона — своя Bradley–Terry MLP-голова │
   │        logit = s(f₁) − s(f₂)  (антисимметричный ранкинг)   │
   │  [CPU] LightGBM на сравнительных фичах пары + метаданные   │
   │        [f₁−f₂, f₁·f₂, |f₁−f₂|, cos, формат/размер байтов]  │
   └────────────────────────────────────────────────────────────┘
        │
        ▼  блендинг предсказаний по OOF (ранговая шкала) → submission.csv
```

**Ключевые решения:**

1. **Замороженные бэкбоны** (`timm`) — устойчиво на маленьком train (8710 пар); не переобучаем ViT с нуля.
2. **Ансамбль разных «взглядов»** — CNN (ConvNeXt-V2 @384) + self-supervised (DINOv2) + CLIP (семантика/«вкус», как в PickScore/HPS).
3. **Bradley–Terry** — общий скор `s(f)` на картинку, логит пары = разность; swap-аугментация и multi-seed снижают дисперсию.
4. **Блендинг отдельных голов**, а не слепая конкатенация всех эмбеддингов — слабые бэкбоны получают малый вес, сильные (CLIP) — большой.
5. **Метаданные** (формат `image_1`, log-размеры байтов) — дешёвый ортогональный сигнал для GBDT.
6. **Извлечение по одной колонке за GPU-сессию** (image_1 → restart → image_2) — обход OOM на Kaggle при parquet с одним row-group.

**Использованные бэкбоны (извлечение фич):**

| Бэкбон | Разрешение | Роль |
|---|---|---|
| `convnextv2_base` | 384 | текстура, деталь, артефакты генерации |
| `vit_large_patch14_dinov2` | 224 | структура, геометрия |
| `vit_large_patch14_clip_224` | 224 | семантика, preference-сигнал (сильнейший одиночный) |

## Результаты

| Итерация | Что добавили | Public LB |
|---|---|---|
| Бейзлайн | один ConvNeXt-V2 @384 + BT + LightGBM | 0.6748 |
| Ансамбль | + DINOv2 + CLIP ViT-L, блендинг голов | 0.6876 |
| Тюнинг | метаданные в GBDT, multi-seed BT, OOF-блендинг | **0.6876+ (2-е место)** |

Стабильный паттерн: рост **OOF AUC** транслируется в рост публичного скора, поэтому OOF —
основной ориентир для решений. Сильнейший одиночный бэкбон — **CLIP** (логично: preference-модели
исторически строятся на CLIP).

## Заключение и идеи для улучшения

Мы прошли путь от бейзлайна на одном бэкбоне (0.675) до ансамбля, занявшего 2-е место (0.688),
за счёт разнообразия представлений и аккуратного блендинга по OOF. Что можно сделать дальше,
примерно в порядке ожидаемой отдачи:

1. **Более сильные / крупные бэкбоны.** CLIP тащит лучше всех → дать ему больше пикселей:
   `CLIP ViT-L @336`, а также **SigLIP @384** и **EVA-CLIP** — топовые экстракторы для
   «вкусовых» задач. Сигнал качества живёт в деталях, поэтому рост разрешения обычно даёт
   больше, чем ещё один похожий бэкбон.
2. **End-to-end файнтюн** (высокий потолок). Сиамский CLIP/ConvNeXt с Bradley–Terry лоссом
   вместо frozen-фич: LoRA или разморозка верхних блоков + layer-wise LR decay. Дороже по GPU,
   но потенциально самый большой прыжок.
3. **Специализированные preference-скоры как фичи.** Готовые **PickScore / ImageReward / HPSv2**
   и **LAION aesthetic predictor** — почти бесплатные ортогональные признаки (скор на картинку →
   разность в паре для GBDT).
4. **Богаче голова и CV.** CatBoost третьей моделью в блендинг; 10-fold вместо 5;
   больше сидов; калибровка/стэкинг вместо линейного rank-blend; `gbdt_pair_mode="full"`.
5. **Сильнее TTA.** Помимо hflip — multi-crop / multi-scale на этапе извлечения фич
   (усреднение эмбеддингов центра и углов).
6. **Псевдо-лейблинг.** Уверенные предсказания на test добавить в train — на маленьком датасете
   это иногда заметно помогает.
7. **Работа с асимметрией слотов.** `image_2` всегда WEBP; аккуратнее моделировать
   позиционный приор и артефакты кодеков (сейчас частично закрыто swap-аугментацией и мета-фичами).
